# 0022. The answerer's trigger forks on compute ownership; cloud generation runs in an ephemeral queue consumer, not a held-open kickoff

- **Status:** Proposed
- **Date:** 2026-06-18
- **Relates:** [ADR-0021](0021-a-conversation-has-one-transport-and-two-triggers.md) (refines its trigger model and resolves its open question on billing-finalize location), [ADR-0017](0017-durable-storage-is-one-per-person-coordination-box.md) (the app-blind anchor this keeps blind), [ADR-0014](0014-an-always-on-reaction-runs-app-semantics-beside-the-app-blind-anchor.md)/[0015](0015-agent-conversations-are-durable-child-docs-answered-by-reactions.md) (reactions), [ADR-0020](0020-answer-bodies-are-native-parts-arrays-streamed-into-y-text.md) (the interrupted-artifact window crash recovery rides)

## Context

ADR-0021 settled "one transport (the doc), two triggers (kickoff vs ambient)" but left the cloud kickoff's *execution model* implicit: today it holds the browser's HTTP request open for the entire generation (`runDocGeneration` + `waitUntil`). Two facts force a refinement. First, Cloudflare's billing is asymmetric: a Durable Object bills for resident wall-clock time even while idle-awaiting I/O ($12.50 per million GB-seconds at 128 MB), while a Worker bills only CPU and *not* I/O-wait duration. So "make the cloud ambient like a daemon" (a resident DO observing the doc) is the expensive shape, and it also forces conversation semantics into the app-blind relay (ADR-0017), breaking the blindness the trust model (ADR-0004) depends on. Second, the held-open kickoff ties the generation's life to the browser tab: close the tab and the answer dies, which forfeits the durable-mailbox property the doc transport was supposed to buy. The deeper truth underneath both: *listening for a turn is free where the owner already runs the machine 24/7 (a self-host daemon, a browser tab) and metered where Epicenter rents the machine.*

## Decision

The answerer's trigger forks on exactly one axis — **who owns the compute the loop runs on** — and both the wake mechanism and the doc-sink substrate are *derived* from it, never chosen per app.

- **Owned compute** (a self-host daemon, a browser tab) hosts a **resident listener** that observes the live doc and syncs natively. Listening costs nothing because the owner runs the machine anyway. This is `attachChatReaction` (daemon) and `attachChatBrowserAnswerer` (tab), already built.
- **Rented compute** (Epicenter cloud) hosts **nothing resident**. The client that wrote the turn fires a **short, stateless kickoff** that (a) reserves credits synchronously — the 402 / auth / rate-limit boundary ADR-0021 keeps — and (b) enqueues a generation job, then returns in milliseconds. An **ephemeral queue-consumer Worker** runs the shared `streamAnswer` core against a hydrated replica and forwards updates to the room by RPC. It is billed as a Worker (no I/O-wait duration charge), runs up to the queue-consumer wall-clock ceiling, and is **independent of the client connection**.

The app-blind relay (ADR-0017) never reads conversation semantics: chat-awareness lives only in the chat-aware kickoff route and the consumer, never in the room Durable Object. Billing reconciles **in the consumer** (this resolves ADR-0021's open question on finalize location): reserve at the kickoff keyed to the reply's `generationId` (ADR-0021's bill-at-the-claim), and **confirm the reservation at completion** (release it on a no-op or pre-stream error). There is no per-token metering; reservation is a fixed per-model credit lock, so "finalize" is confirm/release, not a token settlement.

A consequence of "the kickoff is a rented-compute concept" is structural: **the doc-generation HTTP route is not a shared-library responsibility.** Verified against the code, the "hydrate a replica and forward updates over the room's public RPC" mechanism (`runDocGeneration`) has a single production caller — the cloud doc route — while the owned-compute answerers (`attachChatReaction`, `attachChatBrowserAnswerer`) drive a local `Y.Doc` in-process and never touch it. So the route, the Cloudflare Queue, and the consumer all live in the **cloud deployment (apps/api)**, not in `packages/server`. The library mounts only the SSE route (interactive, client-shared) and exports `runDocGeneration` as a billing-free mechanism the cloud consumer imports. This keeps the library billing- and infra-agnostic without an injection seam, and it strengthens ADR-0017: no request handler in the shared library reads conversation semantics at all.

This drops one capability deliberately: **self-host server-side doc generation.** A self-host worker that held the request open to generate was a pre-decision vestige; under this ADR, owned compute answers ambiently with a resident daemon, so self-host mounts SSE only and answers via the daemon. A self-hoster who wants server-rendered doc answers runs the daemon (free on their hardware) — the same asymmetry this whole decision exploits. Self-host is community-supported, so this clean break is acceptable.

Every runtime still runs the one `streamAnswer` core; only the wake and the sink substrate differ, and both follow from the one ownership axis. Two further trade-offs follow from "rented compute hosts nothing resident and the generation is detached from the client": **cloud has no user-initiated stop** (the consumer never reads the doc back mid-generation and the client cannot reach it — the durable-mailbox property is "you wrote the turn, you get the answer"; a crash still decays the artifact to "interrupted" via ADR-0020), and **cloud doc generation is house-key only** (BYOK is not carried through the queue, so no user key sits at rest in a queue message; a BYOK doc request is refused at the kickoff). BYOK on the interactive SSE route is untouched.

## Consequences

- The cloud kickoff stops holding a multi-minute HTTP connection; it becomes reserve + enqueue + return. A cloud generation now **survives the tab closing** — the durable mailbox, for cloud and not just daemons.
- **No Durable Object pays resident duration to generate.** The room DO stays a content-agnostic relay and never gains chat code; the generation's I/O-wait runs in a Worker, which CF does not bill for duration.
- `AgentConfig.runtime` stops being a chat concept and becomes a **compute-site** property. The browser reads one honest bit: is this agent answered by a resident listener (do nothing) or by rented/ephemeral compute (fire the short kickoff)?
- A **Queue + a consumer Worker** enter the stack **in apps/api**; `runDocGeneration` is **re-homed** (its library-route caller is removed; the cloud consumer imports the still-exported mechanism) and **re-triggered** (HTTP route → queue message), not rewritten. The reservation lock (`apps/api/worker/billing/service.ts`) keys to `generationId` and gains a serializable finalize the consumer calls; only *where* confirm/release runs moves (route middleware → consumer).
- **The shared library loses its doc-generation HTTP route.** `packages/server` becomes relay + auth + ownership + SSE. The worker entry becomes `{ fetch, queue }`. Self-host mounts SSE only and loses server-side doc-gen (answers with a daemon).
- Crash and idempotency are already covered by existing mechanisms: a consumer crash leaves an interrupted artifact (ADR-0020's active-generation window decays it to "interrupted"); queue redelivery is a safe no-op against the existence-is-the-claim check; a client that died before its kickoff completed re-pokes the unanswered turns it owns on next load.
- **Forecloses** making the cloud a pure ambient reaction (refused for the residency cost *and* the blind-relay break) and synchronous server-run browser-tool chat at SSE latency (already ADR-0021's trade). A self-hoster who wants ambient, always-listening answering runs their own resident daemon — free on their hardware, which is the exact asymmetry this decision exploits.

## Considered alternatives

- **Ambient cloud — a resident DO answers from sync alone (the γ option).** Rejected: pays CF duration for residency, and forces conversation semantics into the app-blind relay.
- **In-DO generation triggered by a short kickoff.** Rejected: still pays DO duration (~$0.0002 per 2-minute generation, pure overhead) that a queue-consumer Worker avoids entirely.
- **Keep the held-open kickoff (today's α).** Rejected: the generation dies when the browser disconnects; no durable mailbox for cloud answers.
- **Direct service-binding fetch from the kickoff instead of a queue.** Rejected: the generation must outlive the kickoff request and `waitUntil` caps at 30 s; the queue is the durable handoff to an execution context independent of the request.
- **Keep the doc-generation route in the shared library and inject the cloud reserve+enqueue as a strategy (or mount a parallel cloud route).** Rejected: both keep a route the library should not host. The doc-gen route has one production caller (the cloud) and the owned-compute answerers never use it, so the route is a rented-compute concept. Removing it from the library is subtractive and needs no injection seam; the library keeps only the reusable `runDocGeneration` mechanism.
- **An out-of-band cancel so cloud keeps a stop button.** Rejected for now: the consumer never reads the doc back mid-generation (single-writer, snapshot-once), so a cancel would mean a new channel into a detached worker. The durable-mailbox property makes "always completes" the honest default; revisit only if product needs interrupt.
