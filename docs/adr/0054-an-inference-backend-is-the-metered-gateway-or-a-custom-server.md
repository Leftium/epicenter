# 0054. An inference backend is either the metered Epicenter gateway or a custom OpenAI-compatible server

- **Date:** 2026-06-22

## Context

The inference engine is swappable by base URL ([ADR-0050](0050-the-inference-contract-is-openai-compatible.md)), but every chat app nails it to the metered Epicenter gateway: `createOpenAiAgentEngine({ fetch: auth.fetch, baseURL: <Epicenter /v1> })`. The swap point exists and is unreachable, so [ADR-0049](0049-inference-is-its-own-box-the-daemon-never-infers.md)'s promise (point at a local Ollama, a self-hosted gateway, OpenRouter, or your own key) is undelivered. Making it real means the backend is a user setting, and the setting cannot be a free URL string, because the base URL, the auth, the key, and the valid model ids are coupled. The Epicenter bearer is only for Epicenter ([ADR-0053](0053-the-epicenter-bearer-is-an-audience-scoped-credential.md)); a local Ollama needs no auth; a third party needs the user's key; and a model id only means something against the backend that serves it.

## Decision

A chat app's inference backend is a user setting with two modes, resolved per turn.

- **Hosted (default).** The Epicenter gateway. `auth.fetch` carries the bearer (scoped to Epicenter by [ADR-0053](0053-the-epicenter-bearer-is-an-audience-scoped-credential.md)), credits are metered, and the model is the curated Fast/Best catalog. The gateway is house-key-only: it never accepts a provider key in the request body.
- **Custom.** A user base URL with a plain fetch and an optional `Authorization: Bearer <key>`, never the Epicenter bearer, no Epicenter metering, and a free-text model id. This is where "bring your own key" lives: your key on your box (a local Ollama, a self-hosted gateway, OpenRouter), never handed to Epicenter.
- **Config and secret are split**, the way Zed and VS Code split provider config from the keychain. The custom base URL, model, and key are device-local in every app (`chrome.storage.local` in the extension, `localStorage` in the web apps), never the synced workspace. A `localhost` URL and an API key are device-scoped: a synced one is a secret on the plaintext relay ([ADR-0004](0004-trust-the-relay-reject-zero-knowledge.md)) and a URL that means nothing on another device. This follows the precedent in `specs/20260612T091000-whispering-custom-backend-profiles.md` (identity may sync, keys stay device-local). A synced "global" preference layer is a clean later addition; the key stays device-local regardless.
- **The model travels with the backend.** The per-turn resolver returns the backend and a model valid for it. A stored model the active backend cannot serve falls back to that backend's default rather than being sent and rejected, so switching to Ollama never ships `gpt-5.5` and switching back never poisons a thread with an unknown id. A backend change takes effect on the next turn: the turn snapshots its backend once, and the picker is disabled while a turn generates, so a transcript never spans backends mid-tool-round.

## Consequences

- **BYOK collapses to one path.** The gateway's body-`apiKey` passthrough and its metering bypass are deleted (the `packages/server` inference route, the apps/api billing policy, and their two tests), so the Epicenter gateway provably never receives a provider key. This amends [ADR-0049](0049-inference-is-its-own-box-the-daemon-never-infers.md) ("Epicenter's gateway when passed through" is withdrawn) and [ADR-0050](0050-the-inference-contract-is-openai-compatible.md) ("BYOK passthrough" struck from the gateway's responsibilities).
- **The gateway becomes a pure passthrough proxy.** The client owns OpenAI-SSE normalization, and it has to, because custom mode reaches a provider directly and never touches the gateway. So the gateway's Gemini tool-call index injection is deleted as redundant: the client already accumulates index-less deltas, in both hosted and custom paths. Provider quirks are normalized in exactly one place, the client; the gateway only authenticates, meters, injects the house key, and forwards bytes. This also amends [ADR-0050](0050-the-inference-contract-is-openai-compatible.md), which had the gateway own provider normalization.
- **Self-host changes posture.** House keys stay optional, but a keyless self-host gateway now serves nobody (it was already non-functional, because no client ever sent a body key). The self-host story becomes: set a house key for the providers you serve, or point clients at your own inference server.
- **A web-app user who wants their own key with no Epicenter billing must run a gateway or use the extension.** A browser cannot reach a raw provider API cross-origin (CORS), and the hosted gateway is always metered, so this own-key-no-billing path closes for the web apps. The extension and a desktop shell can custom-direct freely. A narrow, named loss, accepted for the guarantee that the gateway never holds your key.
- **Custom model selection is free-text first.** `/v1/models` auto-discovery is deferred as an Ollama-style affordance for known-local backends. Even Zed requires you to declare models for a generic OpenAI-compatible backend, so a free-text id is the honest floor, not a gap; a deployed web app hits the same CORS wall on `/v1/models` as on completions, so discovery only helps where chat already works.
- **The loop gains a max-step guard.** Its tool loop was unbounded; a confused backend could spin it forever. The engine's per-turn read resolves backend and model together, so the two can never drift into an illegal pair.
- **All three chat apps get custom mode.** Vocab, capability-free with a hosted-only default model, gains a custom model field so it can point at a local model; its hosted default stays the Chinese-tuned catalog model.

## Considered alternatives

- **A third "Hosted plus BYOK" mode** (your key through the Epicenter gateway, metering bypassed). Rejected: it is not a distinct backend, it is the hosted pair plus a body field, so it muddies the resolver and hands Epicenter a provider key for nothing that custom-direct does not already give you. Dropping it is what makes the gateway provably key-free.
- **Backend per conversation, like the model.** Rejected: a backend is a device-scoped resource and a conversation is portable synced content; pinning a backend to a synced conversation would sync a secret and a URL that is meaningless elsewhere. Backend is per device; the model stays per conversation.
- **Fix the backend at engine construction and rebuild the handle on change.** Viable, and it has none of the per-step hazards. The per-turn resolve is chosen because it is symmetric with how the model already updates and avoids tearing down conversation handles on a settings change; the per-turn snapshot plus the generating-lock buys the same safety.
- **Lift the backend into the per-step read (the naive "runtime-editable").** Rejected: a mid-tool-round switch would send the new backend a transcript claiming it issued the previous backend's tool calls, corrupting the message array. The backend is snapshotted per turn instead.
