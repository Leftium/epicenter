# BGSW AI Runtime — Move AI from Hub to Chrome Extension Background Service Worker

**Date**: 2026-02-23
**Status**: Draft (POC validated)
**Author**: AI-assisted design session

---

## Overview

Move the AI runtime (`chat()` from `@tanstack/ai`) from the hub server into the Chrome MV3 background service worker (BGSW). Tools execute directly against Chrome APIs instead of writing to a command queue. Streaming happens via Y.Doc + BroadcastChannel between BGSW and side panel. The hub server retains `/ai/chat` as a generic endpoint for future apps, the `/proxy/:provider/*` reverse proxy for operator-key API access, and the existing auth + sync relay. Only tab-manager-specific code (hardcoded system prompt, tools, room name) is removed from the hub.

---

## Motivation

### Current State

The AI chat flows through the hub server:

```
Side Panel → SSE POST /ai/chat → Hub Server → chat() + tools → SSE stream back
                                        ↓
                                  Mutation tools write to commands table
                                        ↓
                                  Y.Doc syncs command to BGSW
                                        ↓
                                  BGSW executes Chrome APIs
                                        ↓
                                  Result syncs back through Y.Doc to server
                                        ↓
                                  Server resolves tool promise, continues chat()
```

The hub server plugin (`packages/server/src/ai/plugin.ts`) hardcodes:
- Tab-manager system prompt
- `getDoc('tab-manager')` room name
- 13 tab-manager-specific tools (5 read, 8 mutation)

Mutation tools use a command queue: server writes a command row → Y.Doc syncs to BGSW → BGSW executes Chrome API → writes result → syncs back → server's `waitForCommandResult()` resolves. This adds 2 sync round-trips (~200-500ms each) per mutation.

### Problems

1. **Hub server is tab-manager-specific**: The AI plugin is coupled to one app. A second app using the hub would need its own hardcoded plugin. The hub should be generic infrastructure.
2. **Command queue adds latency and complexity**: Every mutation takes 2 Y.Doc sync round-trips through WebSocket. `waitForCommandResult()` polls with a 30-second TTL. If the device is offline, the command times out silently.
3. **Tools can't use Chrome APIs directly**: The server has no `browser.tabs.*` access. Read tools query Y.Doc (which works), but mutation tools must proxy through the command queue. The BGSW already has both Y.Doc and Chrome API access.
4. **API keys route through the hub**: The user's API key travels from extension → hub server → provider. With BGSW-side `chat()`, BYOK keys go directly to the provider — shorter path, one fewer trust boundary.
5. **Hub cannot have arbitrary code execution**: The hub is meant to be shared infrastructure. Running app-specific tool logic there violates the separation of concerns.

### Desired State
```
Side Panel ←—BroadcastChannel—→ BGSW (runs chat(), tools call Chrome APIs directly)
                                   ↓ fetch() with dangerouslyAllowBrowser (BYOK)
                                   or
                                   ↓ fetch() with baseURL → Hub /proxy/:provider/*
                                 Provider APIs (Anthropic, OpenAI, etc.)
Hub Server = auth + sync relay + generic /ai/chat + /proxy/:provider/*
             (no tab-manager-specific code — system prompt, tools, room name removed)
```

User types "close my YouTube tabs" → BGSW's `chat()` calls `searchTabs` (queries local Y.Doc) → finds 5 YouTube tabs → calls `closeTabs` (directly calls `browser.tabs.remove()`) → responds "Closed 5 YouTube tabs." No command queue. No server round-trip for tool execution.

### Auth Flow (Already Logged-In User)

```
1. User signs in via hub (Better Auth) → session token stored
2. Extension stores session token in chrome.storage
3. Local server validates token against hub (GET /auth/get-session)
   → cached 5 min via createHubSessionValidator()
4. Hub /proxy/:provider/* also validates session token before proxying
   → same pattern: Bearer token → validate → inject real API key → forward
```

### Hub URL Configuration

The extension stores two URLs in `chrome.storage.local` (`settings.ts`):

- **`serverUrl`** — local server (default `http://127.0.0.1:3913`)
- **`hubServerUrl`** — hub server (default `http://127.0.0.1:3913`, overridable to e.g. `https://hub.epicenter.so`)

In single-server dev setups both point to the same address. In production, `hubServerUrl` points to the remote hub. The BGSW reads `hubServerUrl` when constructing the operator proxy `baseURL` for adapters.

### What This Spec Changes vs What Stays

| Component | Before | After |
| --- | --- | --- |
| Hub `/ai/chat` | Tab-manager-specific (hardcoded prompt, tools, room) | **Stays as generic endpoint** — remove tab-manager coupling, keep for future apps |
| Hub `/proxy/:provider/*` | Exists, used by OpenCode | **Stays unchanged** — BGSW uses it for non-BYOK users via adapter `baseURL` |
| Hub auth (`/auth/*`) | Better Auth with bearer plugin | **Stays unchanged** |
| Hub sync relay (`/rooms/*`) | Y.Doc relay | **Stays unchanged** |
| Local server | Workspace CRUD + sync | **Stays unchanged** |
| Side panel → hub SSE | `fetchServerSentEvents()` to `/ai/chat` | **Replaced** by `chrome.runtime.sendMessage` to BGSW |
| BGSW command queue | Consumer executes Chrome APIs from server commands | **Removed** — tools call Chrome APIs directly |
| `waitForCommandResult()` | Server polls for command completion via Y.Doc | **Removed** — no command queue means no waiting |
| `commands` table | Y.Doc table for server→BGSW command passing | **Removed** (separate PR, breaking schema change) |

| Test | Result | Notes |

### TanStack AI in Browser Contexts

- `chat()` is isomorphic — designed for edge runtimes (Cloudflare Workers, Vercel Edge), same constraints as service workers
- All adapters accept `baseURL` config for proxying: `createAnthropicChat(model, apiKey, { baseURL: 'https://proxy.example.com' })`
- `createAnthropicChat(model, apiKey, config?)` — model first, apiKey second, returns adapter directly (not curried)
- Stream chunks use AG-UI protocol: `RUN_STARTED`, `TEXT_MESSAGE_CONTENT` (with `.delta`), `TOOL_CALL_START`, etc.
- `agentLoopStrategy: maxIterations(n)` controls tool-call loop depth

### Browser API Key Security

| Scenario | Approach | `dangerouslyAllowBrowser` | Risk |
| --- | --- | --- | --- |
| BYOK (user's own key) | Direct to provider from BGSW | Required | None — user's own key, sandboxed extension context |
| Operator-provided key | Cloudflare Worker proxy with `baseURL` | Not needed | Key stays server-side, never exposed to client |
| Hybrid | Try BYOK first, fallback to proxy | Conditional | Best UX — works offline with own key, operator subsidizes if available |

### Precedent: AI in Chrome Extensions

- **AgentBoard** (`igrigorik/AgentBoard`): Runs AI chat in BGSW, streams via `chrome.runtime.connect()` ports
- **web-llm** (`mlc-ai/web-llm`): Runs entire ML engine in extension service worker via `ExtensionServiceWorkerMLCEngineHandler`
- Multiple ChatGPT sidebar extensions call provider APIs directly from extension contexts

### BroadcastChannel for Y.Doc Sync

- BGSW and side panel share origin (`chrome-extension://<id>`) — same IndexedDB, same BroadcastChannel namespace
- `y-indexeddb` does NOT provide real-time cross-context sync (only loads on init)
- BroadcastChannel provides sub-millisecond local sync — implemented as ~15 lines in `broadcast-channel.ts`
- Both contexts (BGSW + popup) run `createBroadcastChannelSync(doc, 'tab-manager')` to stay in sync
- Y.Doc CRDTs handle duplicate updates gracefully (idempotent merges)

## Design Decisions

| Decision | Choice | Rationale |
| --- | --- | --- |
| Where chat() runs | BGSW | Has Chrome API access + Y.Doc + long-lived. Side panel is ephemeral — tools would die when user closes it. |
| Streaming transport | Y.Doc + BroadcastChannel | Already have Y.Doc infra. BroadcastChannel gives sub-ms local sync. No new protocol needed. |
| Tool execution model | Direct Chrome API calls | Eliminates command queue (2 sync round-trips), reduces latency from ~1s to <10ms per tool call |
| BYOK browser access | `dangerouslyAllowBrowser: true` | Anthropic SDK requires it for browser contexts. Safe in extension — user's own key, sandboxed context. |
| Operator key proxy | Cloudflare Worker with `baseURL` | Key stays server-side. Extension sets `baseURL` on the adapter — same API, different endpoint. |
| Hub server AI code | Keep `/ai/chat` generic, remove tab-manager coupling | `/ai/chat` stays for future apps that want server-side chat. Tab-manager-specific system prompt, tools, and `getDoc('tab-manager')` removed. `/proxy/:provider/*` stays unchanged — BGSW uses it via adapter `baseURL`. |
| Tool definitions | Keep in shared package | `toolDefinition()` contracts stay in a shared location. `.server()` implementations move to BGSW. |
| Command queue | Remove (separate PR) | No longer needed when tools execute directly in BGSW. Eliminates `commands` table, consumer, wait-for-result. Breaking schema change — split into its own PR. |
| Read tools in BGSW | Query local Y.Doc | BGSW has the authoritative Y.Doc with all synced devices' data. Same pattern as server read tools. |
| Chat state management | BGSW owns chat lifecycle, side panel observes Y.Doc | BGSW runs `chat()`, writes messages to Y.Doc tables. Side panel reads via table observers + BroadcastChannel. |

### Message Flow: User Sends Chat Message

```
STEP 1: User types message in side panel
─────────────────────────────────────────
Side panel writes user message to chatMessages table in Y.Doc.
BroadcastChannel instantly syncs to BGSW's Y.Doc.

STEP 2: Side panel triggers chat via chrome.runtime.sendMessage
──────────────────────────────────────────────────────────────────
Side panel sends { type: 'chat', conversationId, ... } to BGSW.
BGSW receives message and starts chat() with current conversation context.

STEP 3: BGSW runs chat() with tools
──────────────────────────────────────
chat() calls provider API via fetch().
Streaming chunks arrive as AG-UI events.
For each TEXT_MESSAGE_CONTENT, BGSW appends to a Y.Text or updates
the chatMessages table entry. BroadcastChannel syncs to side panel.

STEP 4: Tool calls execute directly
─────────────────────────────────────
If the model calls searchTabs → BGSW queries Y.Doc tables directly.
If the model calls closeTabs → BGSW calls browser.tabs.remove() directly.
Result returned to chat() inline — no command queue, no sync round-trip.

STEP 5: Response complete
──────────────────────────
chat() finishes. Final message written to chatMessages.
Side panel Y.Doc observer triggers UI update.
```

## Implementation Plan

### Phase 1: BGSW Chat Engine (Core)

Move `chat()` and tool execution from the hub server into the BGSW.

- [ ] **1.1** Create `apps/tab-manager/src/lib/ai/engine.ts` — the chat engine that wraps `chat()` with adapter creation, tool binding, and message management
- [ ] **1.2** Move tool `.server()` implementations from `packages/server/src/ai/tools/read-tools.ts` into the BGSW context (e.g., `apps/tab-manager/src/lib/ai/tools/read-tools.ts`). These are identical — they query Y.Doc tables.
- [ ] **1.3** Rewrite mutation tools as direct Chrome API calls in `apps/tab-manager/src/lib/ai/tools/mutation-tools.ts`. Instead of writing to a commands table, call `browser.tabs.remove()`, `browser.tabs.create()`, etc. directly. Reuse existing action implementations from `apps/tab-manager/src/lib/commands/actions.ts`.
- [ ] **1.4** Wire up `chrome.runtime.onMessage` handler in `background.ts` to receive chat requests from the side panel and start the engine
- [ ] **1.5** Create adapter factory in BGSW with `dangerouslyAllowBrowser: true` for BYOK and `baseURL` support for operator proxy

### Phase 2: Streaming via Y.Doc + BroadcastChannel

Replace SSE streaming with Y.Doc table writes + BroadcastChannel.

- [ ] **2.1** Define the streaming contract: BGSW writes assistant message chunks to `chatMessages` table (progressive Y.Doc updates). Side panel observes via existing table observers.
- [ ] **2.2** Evaluate Y.Text for token-level streaming (each token appended to a Y.Text field, side panel reads `.toString()` on observe). Alternative: write full accumulated text on each chunk (simpler but more Y.Doc traffic).
- [ ] **2.3** Update side panel chat state (`chat.svelte.ts`) to read messages from Y.Doc observers instead of SSE `fetchServerSentEvents`. Remove hub URL dependency.
- [ ] **2.4** Handle tool call state in Y.Doc (tool calls in progress, tool results) so the UI can show tool execution status.

### Phase 3: Decouple Hub AI from Tab Manager

Remove tab-manager-specific code from the hub's AI endpoint. Keep `/ai/chat` as generic infrastructure.

- [ ] **3.1** Refactor `packages/server/src/ai/plugin.ts` — remove the hardcoded tab-manager system prompt, `getDoc('tab-manager')` room name, and tab-manager-specific tool creation. The endpoint stays as a generic `POST /ai/chat` that accepts `messages`, `provider`, `model`, and optional `tools`/`systemPrompt` from the request body. Future apps that want server-side chat can use it directly.
- [ ] **3.2** Remove `packages/server/src/ai/tools/read-tools.ts` and `mutation-tools.ts` — these are tab-manager-specific. The generic endpoint no longer creates tools itself; callers provide them or it runs tool-less.
- [ ] **3.3** Remove `packages/server/src/ai/tools/wait-for-result.ts` — command queue polling, no longer needed.
- [ ] **3.4** Keep `packages/server/src/ai/tools/definitions.ts` if any definitions are reusable, otherwise move to `apps/tab-manager/`.
- [ ] **3.5** Keep `@tanstack/ai` in `packages/server/package.json` — the generic `/ai/chat` endpoint still uses it. Remove only tab-manager-specific provider packages if any.
- [ ] **3.6** Keep `packages/server/src/proxy/plugin.ts` mounted on the hub — BGSW uses it for non-BYOK users.

### Phase 3b: Remove Command Queue (Separate PR)

The command queue removal is a breaking schema change. Split into its own PR for clean rollback.

- [ ] **3b.1** Remove `apps/tab-manager/src/lib/commands/consumer.ts` — command queue consumer
- [ ] **3b.2** Remove `apps/tab-manager/src/lib/commands/constants.ts` — command TTL constant
- [ ] **3b.3** Keep `apps/tab-manager/src/lib/commands/actions.ts` — Chrome API action implementations (reused by BGSW mutation tools)
- [ ] **3b.4** Remove the `commands` table from the workspace definition (breaking schema change — requires migration)

### Phase 4: Operator Key Proxy via Hub (Already Built)

The hub's `/proxy/:provider/*` endpoint (`proxy/plugin.ts`) already does this:

- [ ] **4.1** BGSW detects whether user has BYOK key or should use operator proxy
- [ ] **4.2** For operator proxy: set adapter `baseURL` to `{hubServerUrl}/proxy/{provider}` — the hub validates the session token (same Better Auth token already stored in `chrome.storage`), injects the real API key from env vars, and forwards to the provider
- [ ] **4.3** For BYOK: adapter goes direct to provider with `dangerouslyAllowBrowser: true`
- [ ] **4.4** (Future) Add rate limiting and usage tracking to the proxy if needed

## Edge Cases

### Side panel closed during streaming

1. User sends message, BGSW starts `chat()` streaming
2. User closes side panel mid-stream
3. BGSW continues streaming — it's a long-lived service worker, independent of the side panel lifecycle
4. When user reopens side panel, it reads the completed (or still-streaming) message from Y.Doc via BroadcastChannel sync

### BGSW terminated mid-stream

1. Chrome terminates the service worker (idle timeout, memory pressure)
2. The `chat()` call and any in-flight tool calls are lost
3. On BGSW restart, the partial message remains in Y.Doc (last written chunk)
4. The side panel shows the partial response. User can retry.
5. Mitigation: the offscreen keepalive mechanism already extends BGSW lifetime during active operations

### Multi-device tool calls

1. User says "close YouTube tabs on my work laptop" from their phone
2. BGSW on the phone has Y.Doc with all devices' tabs (read tools work)
3. But `browser.tabs.remove()` can only close tabs on the current device
4. For cross-device mutations, the command queue pattern is still needed — but only for remote device targeting
5. **Recommendation**: Phase 1 supports local-device mutations only. Cross-device mutations are a separate feature that could reintroduce a targeted command queue or use a different signaling mechanism.

### No API key configured

1. User opens chat, no BYOK key set, no operator proxy configured
2. BGSW sends error response: `{ type: 'error', message: 'No API key configured' }`
3. Side panel shows settings prompt to add an API key

### Provider rate limit or error

1. `chat()` stream yields `RUN_ERROR` event with error details
2. BGSW writes error to chatMessages table
3. Side panel displays error inline in the conversation

## Open Questions

1. **Token-level streaming: Y.Text vs progressive table writes?**
   - Y.Text: each token appended, side panel reads `.toString()` — minimal overhead, natural for streaming text
   - Progressive writes: update the full message text on each chunk — simpler code, more Y.Doc traffic
   - **Recommendation**: Test Y.Text in a follow-up POC. Progressive writes are simpler for Phase 1.

2. **Should tool definitions stay in `packages/server/` or move to `apps/tab-manager/`?**
   - Definitions are pure schema contracts (`toolDefinition()` + arktype) — no runtime deps
   - Could live in `@epicenter/tab-manager/ai-tools` export or a new shared location
   - **Recommendation**: Move to `apps/tab-manager/src/lib/ai/tools/definitions.ts` since they're tab-manager-specific. If another app needs tools, it defines its own.

3. **Cross-device mutations: keep a minimal command queue or defer?**
   - The current command queue works for cross-device mutations
   - But it's complex and the BGSW approach eliminates it for local mutations (the 90% case)
   - **Recommendation**: Defer. Remove the command queue in Phase 3b as a separate PR (breaking schema change). Add cross-device mutation support as a separate feature later if needed.

4. **Chat state ownership: who manages conversation lifecycle?**
   - Currently `chat.svelte.ts` in the side panel creates `createChat()` instances and manages state
   - Moving to BGSW means BGSW owns the `chat()` call, but the side panel still needs reactive UI state
   - **Recommendation**: BGSW is the chat engine (runs `chat()`, writes results to Y.Doc). Side panel is pure UI (reads Y.Doc via observers, sends user messages via `chrome.runtime.sendMessage`). The `createChat()` Svelte client may still be useful for managing UI state if it can read from Y.Doc instead of SSE.

5. **Should we keep `@tanstack/ai-svelte`'s `createChat()` or build a simpler Y.Doc-based reactive state?**
   - `createChat()` manages messages, loading state, streaming, tool approval — lots of built-in UI logic
   - But it's designed for SSE client-server patterns, not Y.Doc observation
   - **Recommendation**: Start with a minimal Y.Doc-based approach in Phase 2. If we miss `createChat()` features, evaluate adapting it later.
