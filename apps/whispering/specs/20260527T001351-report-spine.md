## One Sentence

Domain code returns `defineErrors` tagged errors unchanged; one delivery primitive `report.{error,success,info,loading}` accepts a single `Notice` shape (title and description default from the tagged-error's `name` and `message`), and fans events to a `wellcrafted/logger` sink stack (console + memory + toast + OS) under per-level routing rules: only `error` escalates to OS-when-unfocused, only `error` persists, everything else times out.

## Current Shape

```
OpenaiTranscriptionServiceLive.transcribe(blob)
  └─ defineErrors → OpenaiError.FileTooLarge({ sizeMb, maxMb })
       ↓
  dispatchTranscription:
    services.transcriptions.openai.toWhisperingErr(error)   ← per-service sidecar
       ↓
    WhisperingErr({ ...toasts.fileTooLarge(error) })        ← shared copy helper
       ↓                                                      (UI bag + name + severity)
  Result<string, WhisperingError> returns up
       ↓
  pipeline.ts:
    if (err.name === 'WhisperingError') notify.error({ id, ...err })   ← branch A
    else notify.error({ title, description, action: { more-details, error } })   ← branch B
       ↓
  notify.error(options):
    dev:    console.error('[Notify]', options)
    history: notificationLog.addLog(options)
    toast:   services.toast.show(options)         → Sonner
    os:      services.notification.notify(options) → Tauri (loading skipped)
       ↓
  "More details" → moreDetailsDialog.open({ content: error })
```

Four shapes (tagged error, `*Live.toWhisperingErr` output, `toasts.X` helper output, `WhisperingError`/`UnifiedNotificationOptions`). Two branches in every call site that may consume non-Whispering errors. Severity encoded on the data **and** by method choice. Magic auto-injection of `more-details` in `WhisperingErr`. Cross-surface god-bag with fields that no-op on the surface they're not for.

## Target Shape

```
Service: defineErrors → SomeError.Variant({ ... })
   ↓                                       (tagged error, unchanged)
Operation call site:
   if (error) { report.error({ cause: error }); return; }     ← minimum viable error
   ↓
report.error(problem):
   composed sink fans out:
     consoleSink     ← dev visibility
     memorySink      ← diagnostic ring buffer (not a UI)
     toastSink       ← always; auto-renders title=humanize(cause.name), desc=cause.message
     osNotifySink    ← only for level==='error' and !document.hasFocus()
   logger:
     log.error(problem.cause)
```

One shape (`Notice`), one extension (`Problem` = `Notice & { cause: required }`). No translation step: context-specific copy is an inline override at the call site when needed. Routing is a pure function of level + focus.

## The Asymmetric Refusals

These are the things we **refuse to support**. Each refusal collapses substantial complexity.

### Refusal 1: No per-call urgency knob

Level (the verb) is the only loudness control. No `urgency: 'low' | 'normal' | 'high'`, no `important: true`, no `loud: true`. If a thing needs to escalate beyond a level's default routing, you picked the wrong level.

### Refusal 2: No OS notification on success

**Errors escalate to OS when unfocused. Success does not, ever.**

The success of a transcription is confirmed by its effect: text appears at the cursor, file is saved, sound plays. The user already has a signal. An OS notification on success is redundant noise. Failures have no other signal; the OS notification is the only way the user finds out.

This collapses the routing rule from "depends on level + focus + maybe more" to:

```
error    → toast + (OS iff unfocused)
warn     → toast only
success  → toast only
info     → toast only
loading  → toast only
```

One thing fires OS. Everything else is toast.

### Refusal 3: OS notifications are title + body only

No icon configuration on the cross-surface bag. No `silent`, `requireInteraction`, `persist`. No actions on OS notifs.

Tap-to-focus-app is the only OS-side interaction. The toast carries the rich affordances (actions, "more details," in-place updates). The OS surface is the dumb "hey, look over here" tap.

`UnifiedNotificationOptions` collapses from ~10 cross-surface fields to: `{ title, description? }` + toast-only extras (`action`, `id`).

### Refusal 4: No in-app "notification history" UI as a first-class surface

`memorySink` exists for diagnostics (what just happened, in dev tools, for support). It is **not** a notification center the user is expected to visit. No notification log route, no badge, no unread counter. The user sees notifications when they happen, or they don't.

If a user needs to recover an action from a notification they missed, the fix is making the action discoverable from the actual destination (recordings page, settings page), not adding a notification center.

### Refusal 5: One action shape

```ts
type NoticeAction = { label: string; onClick: () => void | Promise<void> };
```

No `link` / `button` / `more-details` variants. A link is a button with `onClick: () => goto(href)`. "More details" is the **implicit default** when a `Problem` has no explicit action. The toast renderer pulls `cause` and renders the modal.

### Refusal 6: Four levels, no `warn`

`error`, `success`, `info`, `loading`. That's it.

Going through every `notify.warning` call site, each one collapses cleanly into either `error` (if it's actually a failure the user should know about) or `info` (if we recovered). The split forces a real call: did it fail or didn't it. There is no "yellow middle." Yellow is a styling preference, not a level.

### Refusal 7: One shape, one optional field, level enforces required-ness

```ts
type Notice = {
  title?: string;                    // auto-derived from cause.name if omitted
  description?: string;              // auto-derived from cause.message if omitted
  action?: NoticeAction;
  cause?: AnyTaggedError;            // optional: enriches log entry if present
};

type Problem = Notice & { cause: AnyTaggedError };  // required: drives More Details default
```

`cause` means one thing at every level: "this event has a tagged-error origin; the logger captures it." The error level adds *one* extra behavior on top: render "More Details" by default. Otherwise the same field, the same semantics, all the way down. `title` and `description` are optional too: `report.error({ cause: err })` is a valid call site.

### Refusal 8: Loading is a handle, not a string id

```ts
const t = report.loading({ title: '📋 Transcribing…' });
const { data, error } = await transcribe();
if (error) { t.reject({ cause: error }); return; }
t.resolve({ title: 'Done', description: data, action: COPY });
```

No threading a `string` `toastId` through five function boundaries. The handle owns the in-place lifecycle. Handle methods: `resolve(Notice)`, `reject(Problem)`, `update(Notice)`, `dismiss()`. No mirror of every level. A loading toast resolves to a result or rejects to a problem, period.

### Refusal 9: Errors persist by default, no `persist` field

The truthy default: `error` toasts stay until dismissed. `success` and `info` time out at 3-5s. `loading` dismisses when resolved/rejected/dismissed. No `persist`, `duration`, or `requireInteraction` knobs on the API. If a thing deserves persistence, it's an error; if it doesn't, it isn't.

### Refusal 10: OS notifications are fire-and-forget, no IDs

With loading toast-only and no in-place OS replacement, every OS notification is fresh. Delete `hashNanoidToNumber`, the active-notification listing, `removeNotificationById`, and all the ID-string-to-int hashing machinery. ~80 LOC of `notifications/index.tauri.ts` evaporates.

## API Sketch

```ts
// ── Shared shapes ───────────────────────────────────────────────────
type NoticeAction = { label: string; onClick: () => void | Promise<void> };

type Notice = {
  title?: string;
  description?: string;
  action?: NoticeAction;
  cause?: AnyTaggedError;
};
type Problem = Notice & { cause: AnyTaggedError };

// ── Top-level spine ─────────────────────────────────────────────────
export const report = {
  error:   (r: Problem) => void,                                  // toast + OS-if-unfocused, persists
  success: (r: Notice)  => void,                                  // toast only, auto-dismiss
  info:    (r: Notice)  => void,                                  // toast only, auto-dismiss
  loading: (r: Notice)  => LoadingHandle,                         // toast only, until resolved
};

// ── Loading handle ──────────────────────────────────────────────────
type LoadingHandle = {
  resolve: (r: Notice)  => void;   // success-styled completion
  reject:  (r: Problem) => void;   // error-styled completion (persists, may OS-notify)
  update:  (r: Notice)  => void;   // re-render same toast, still loading
  dismiss: () => void;
};

// ── Minimum viable call ─────────────────────────────────────────────
// title and description default to humanize(err.name) and err.message:
if (error) { report.error({ cause: error }); return; }

// Inline override only when context-specific copy or an action helps:
if (error) {
  report.error({
    cause: error,
    title: 'Authentication required',
    action: { label: 'Update API key', onClick: () => goto('/settings/transcription') },
  });
  return;
}
```

There is no `presentTranscriptionError` switch. Inline overrides live at the call site, scoped to where the context is. The 90% case has zero override; the 10% case has a 5-line object.

## Routing Rules

One function, expressed once, used by one sink:

```ts
// osNotifySink internals
function shouldFire(level: Level): boolean {
  if (level !== 'error') return false;       // refusal 2: only errors escalate
  return !document.hasFocus();               // refusal 1: focus is the only routing input
}
```

No settings flag. No urgency. No focus subscription bookkeeping: read at emit time. The toast sink fires on every level; the OS sink fires only on error-when-unfocused.

| Level   | Toast      | OS                  | Duration                | Logger        |
|---------|------------|---------------------|-------------------------|---------------|
| error   | red        | iff `!hasFocus()`   | persist until dismissed | `log.error`   |
| success | green      | never               | 3s                      | `log.info`    |
| info    | neutral    | never               | 4s                      | `log.info`    |
| loading | spinner    | never               | until resolved          | `log.info`    |

## Sinks

`wellcrafted/logger` is the spine. The composed sink:

```ts
const reportSink = composeSinks(
  consoleSink,                       // dev visibility (replaces notify.ts switch)
  memorySink({ capacity: 100 }),     // diagnostic ring buffer (NOT a UI history)
  toastSink,                         // every event
  osNotifySink,                      // error + unfocused only
);
const log = createLogger('whispering/report', reportSink);

// Inside report.error/warn, also call log.error/log.warn(problem.cause) to
// route the tagged error through the logger.warn/error pathway natively.
```

`toastSink` reads `event.data` (the Notice/Problem) and:
- Picks Sonner method from `event.level`.
- Default action for `Problem` without explicit `action` = render a "More details" button that opens `moreDetailsDialog` with `cause`.

`osNotifySink` reads `event.data` for `title` + `description` only. Everything else ignored. Tap-to-focus is the only interaction.

## Call Site Examples

### `pipeline.ts` (transcribe path)

```ts
const t = report.loading({ title: '📋 Transcribing…', description: 'Your recording is being transcribed…' });

const { data: text, error } = await transcribeArtifact(artifact);
if (error) { t.reject({ cause: error }); return; }   // ← auto title/description from err

sound.playSoundIfEnabled('transcriptionComplete');
await deliverTranscriptionResult({ text, loading: t });

const { error: saveAudioError } = await saveAudioPromise;
if (saveAudioError) report.error({
  title: 'Audio not saved',
  cause: saveAudioError,
});
```

### `pipeline.ts` (no-transformation case)

Old: `notify.warning(...)`. New: `report.info(...)`: no failure, just an FYI with CTA:

```ts
report.info({
  title: 'No transformation selected',
  description: 'Please select a transformation in settings.',
  action: { label: 'Open settings', onClick: () => goto('/transformations') },
});
```

### `delivery.ts` (cursor write recovered)

Old: `notify.warning(...)`. New: `report.info(...)` because we *recovered* (offered manual copy):

```ts
report.info({
  title: "Couldn't write to cursor",
  cause: writeError,                  // logged, More Details NOT shown by default on info
  action: { label: 'Copy to clipboard', onClick: () => copyToClipboard(text) },
});
```

### `delivery.ts` (copy failed, no recovery)

Old: `notify.warning(...)`. New: `report.error(...)` because the user just lost the copy action:

```ts
if (copyError) report.error({ cause: copyError });   // title="Clipboard write", desc=err.message, More Details auto
```

### Inline override at call site

If a transcription `Unauthorized` deserves a "Go to settings" CTA:

```ts
if (error.name === 'Unauthorized') {
  report.error({
    cause: error,
    title: 'Authentication required',
    action: { label: 'Update API key', onClick: () => goto('/settings/transcription') },
  });
  return;
}
if (error) { report.error({ cause: error }); return; }   // everything else auto
```

## What Gets Deleted

| Concept | Why it goes |
|---|---|
| `WhisperingError` | UI bag wearing an error name |
| `WhisperingErr` factory | Inline overrides at call site or auto-derive |
| `WhisperingResult<T>` | Domain code returns `Result<T, DomainError>` directly |
| `severity` field on errors | Level is the verb |
| `*Live.toWhisperingErr` sidecar (all cloud services) | Inline at call site, only where needed |
| `rpc/transcription-errors/shared.ts` | Folded into call sites or deleted entirely |
| Auto-injection of `more-details` in `WhisperingErr` | Toast renderer default for `Problem` w/o action |
| `MoreDetailsAction` variant | Default behavior, not a variant |
| `LinkAction` / `ButtonAction` variants | Collapse to `{ label, onClick }` |
| `name === 'WhisperingError'` branches (pipeline, delivery) | Single path |
| `notificationLog.addLog` / log route | `memorySink` is diagnostic only |
| Dev `console.X` switch in `notify.ts` | `consoleSink` |
| `UnifiedNotificationOptions` god-bag fields: `icon`, `silent`, `requireInteraction`, `persist`, `variant` | Refusals 3, 9 |
| `variant: 'loading'` on the OS path | Loading is structurally toast-only |
| Threading `toastId: string` through pipeline boundaries | `LoadingHandle` |
| Per-call urgency | Refusal 1 |
| `notify.warning` and `report.warn` | Refusal 6: 4 levels only |
| `hashNanoidToNumber`, `removeNotificationById`, active-notif listing | Refusal 10: fire-and-forget OS |
| `presentTranscriptionError` (or any per-domain `present*` helper) | Auto-derive + inline overrides |

## Decisions (resolved during design)

- **Auto-derive title and description from `cause`.** `humanize(name)` for title, `cause.message` for description. Overrides are inline at the call site, not a translator function.
- **No `present*` helpers, per-domain or global.** Inline overrides at the ~10% of call sites that need context-specific copy. The other 90% is just `report.error({ cause: err })`.
- **`warn` does not exist.** Force the developer to decide: error or info.
- **`info` exists.** Distinct from `success`: success implies completion of an intended action; info is FYI with optional CTA.
- **Sound is not a sink.** Sounds tie to specific events, not generic levels. Keep `sound.playSoundIfEnabled(...)` as a sibling call.
- **Two emission paths**: `report.*` for user-facing events; `log.*` (wellcrafted/logger) for diagnostic-only events that should never surface to a user (analytics failures, background retries). Document the boundary in the report module's JSDoc.

## Open Questions

1. **`humanize` algorithm.** `"PayloadTooLarge"` → `"Payload too large"` is easy. What about edge cases: `"OAuthFlowFailed"` (acronym), `"HTTP500"` (numbers), `"MissingApiKey"` (capitalized acronym mid-name)? Initial implementation: split on uppercase boundaries, lowercase non-first words, special-case all-caps tokens. Don't try to be too clever: variant names are written by us, so we control the input space.

2. **Focus-state read on web.** `document.hasFocus()` returns true when the *tab* has focus, which is louder than "user is looking at the app." Probably fine: toast still renders, browser notif still fires when actually backgrounded.

3. **Browser notification surface in non-Tauri builds.** Today `notifications/index.browser.ts` passes icon/requireInteraction/silent. Under refusal 3 these go away. Confirm we don't visually regress on the web build.

4. **Mac Tauri OS notification UX.** macOS Tauri OS notifs can show banner-style or alert-style (sticky). We don't expose this per-call. If we want stickiness for errors, that's a per-platform default in `osNotifySink`, not a per-call API knob.

5. **`composeSinks` is sequential and unguarded.** If `toastSink` throws, `osNotifySink` doesn't fire. Today's `notify.ts` swallows OS errors with `console.error`. Decision needed: wrap individual sinks for best-effort, or let throws propagate as bugs to fix. Lean toward propagate-as-bug: silent swallowing is what got us here.

6. **Migration order.** Foundation first (report.ts + sinks). Then can the cloud services drop `toWhisperingErr` in parallel? Yes if we keep `WhisperingErr` working until call sites are migrated. Or do we do a single atomic swap? Prefer staged: each PR can land independently.

## Alternatives Considered

- **UI fields on `defineErrors`** (`title`, `action` baked into variants): would let `report.error(err)` work with no presentation step, but bloats domain errors with UI strings, prevents context-specific copy (provider name, settings link), and is just the `WhisperingError` mistake in a new spot. Rejected.

- **Discriminated union by surface** (`{ kind: 'toast' | 'os' | 'both' }`): forces every call site to know transport. Wrong axis. Rejected.

- **Three-level urgency knob** (low/normal/high): premature. Use cases I invented for "high" are actually mis-classified levels. Rejected.

- **Per-event-type user settings** ("notify on transcription complete: yes/no"): paradox of choice, ongoing maintenance. One coarse "OS notifications: All/Errors only/Off" setting at most, if needed.

## Risks

1. **Wide blast radius.** Every transcription service, every operation, every `WhisperingErr` call site, all of `pipeline.ts` and `delivery.ts`. Stage: report module + sinks first (additive); call sites next (one operation at a time); WhisperingError deletion last.

2. **`writeToCursor` return type** today is `Result<void, TextError | WhisperingError>`. Under collapse, narrows to `Result<void, TextError>`. The macOS-permission case becomes a `TextError` variant that callers can choose to override at the call site if they want a CTA. Otherwise auto-derived from the variant name.

3. **Analytics breakage.** `transcribe.ts:172-176` reads `error_title` / `error_description` off the prebaked `WhisperingError`. Under the new design analytics reads tagged error directly (`error.name`, `error.message`, structured fields). Richer data, but event shape changes: downstream dashboards may need updating.

4. **"Errors are quiet on focused" might bite.** A critical error fires while the user is looking at the app → toast appears (persistent under refusal 9), no OS notif. If they dismiss without reading, no second chance. Mitigation: error toasts auto-render `cause.message` in description and a "More details" button; persistent recovery lives at the destination page.

5. **No `warn` level may push too many things to `error`.** Some current `notify.warning` calls might escalate to OS-when-unfocused after collapse. Walk every call site during migration; force the developer to pick.

6. **Auto-derive from `err.name`** assumes variant names are written for human consumption. Audit defineErrors call sites during migration; rename any cryptic variants.

## Review

**Completed**: 2026-05-27
**Branch**: codex/whispering-recorder-refactor

### Summary

The report spine is implemented end to end. Domain services now return tagged
errors directly; user-facing reporting flows through `report.error`,
`report.success`, `report.info`, and `report.loading`; the old
`WhisperingError`, `notify`, toast service, OS notification service, and
notification history UI have been removed.

### Deviations from Spec

- `wellcrafted/logger` only supports diagnostic levels, so the diagnostic
  `log` uses the console and memory sinks only. This prevents `log.info` and
  `log.error` from creating user toasts.
- The migration covered all executable `notify.*` call sites before demolition,
  not only the staged prompt list.
- The notification services were deleted outright because the new OS sinks do
  not need the old service layer.
